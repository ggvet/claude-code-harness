## マーカー凡例

PM ↔ Impl 運用で使用する標準マーカー:

| マーカー | 意味 | 誰が付ける |
|---------|------|-----------|
| `pm:requested` / `pm:依頼中` | PM がタスクを起票し、Impl へ依頼中 | PM |
| `cc:todo` / `cc:TODO` | Impl の未着手タスク | Impl |
| `cc:wip` / `cc:WIP` | Impl（Claude Code）が着手中 | Impl |
| `cc:done` / `cc:完了` | Impl が作業完了し、PM の確認待ち | Impl |
| `pm:approved` / `pm:確認済` | PM が最終確認を完了 | PM |

**状態遷移**: 新規・更新時の正規出力は `pm:requested → cc:todo → cc:wip → cc:done → pm:approved`。既存 `pm:依頼中 → cc:TODO → cc:WIP → cc:完了 → pm:確認済` も read-compatible。

**後方互換**: `cursor:依頼中` / `cursor:確認済` は `pm:依頼中` / `pm:確認済` の同義として扱う（Cursor PM 運用時の表記）。

## Phase 93: Cross-Session Relay 内部実装 — 独立セッション間 即時 push + cross-agent ハンドオフ [P2]

Purpose: 「独立セッション (worktree/session 跨ぎ) 間の自律 relay + ユーザートリガーの 2 者間ハンドオフ + Cursor↔ClaudeCode↔Codex の cross-agent ハンドオフ」を、外部ツール agmsg を install/連携せず **CCH 内部実装**で実現する。核心事実: cross-session messaging は 3 層 (storage / turn 配達 / monitor 配達) に分解でき、CCH は既に storage (`harness_mem signal` / `harness_session_broadcast`) と turn 配達 (`inbox_check.go` PreToolUse pull) を自前で持つ。欠けているのは **monitor push (常駐ポーリング) の 1 層だけ**。agmsg `watch.sh` (MIT) は実証済み設計パターンの参考とし、共有 SQLite を CCH signal storage に差し替えて移植する。これにより agmsg の外殻 (独自 SQLite / team driver / settings.local.json hook) を一切持ち込まずに同等の即時 relay を得る。

設計判断 (内部実装が外部統合より優れる根拠 + 残る構造的制約):
1. **監査盲点が原理的に消える**。agmsg は delivery 設定時に `<project>/.claude/settings.local.json` に hook を自前で書き、CCH guardrail (`go/internal/guardrail/protected_path_policy.go` は `.claude/hooks/` のみ保護) を素通りする。内部実装の配達層は CCH 自身の `.claude-plugin/hooks.json` + R01-R13 を通るため、この構図自体が無くなる。
2. **redaction に乗る**。storage は既存 harness signal を使うため harness-mem の Layer 1-3 redaction がそのまま効く (agmsg 平文 SQLite の untrusted 問題が消える)。受信内容は `inbox_check.go` の untrusted-data 隔離 (構造化信頼フィールド + disclaimer wrap) を踏襲。
3. **即時 push は CC↔CC 限定** (構造的天井、agmsg も同じ)。monitor 常駐ポーリングを回せる Monitor tool を持つのは CC のみ。Codex/Cursor は Monitor 非保有のため即時 push 不可 (agmsg `SKILL.md`: monitor = claude-code only と同一の天井)。
4. **cross-agent は companion turn pull**。CC↔Codex/Cursor のハンドオフは companion が signal inbox を読み書きする turn 配達。ユーザートリガーのハンドオフ用途に合致し、即時 push は不要。
5. **harness-loop と二者択一**。monitor 常駐と harness-loop は Monitor tool を取り合う。同時併用しない契約を明記 (内部実装でも残る制約)。
6. **actas 排他 lock 不要**。agmsg actas は「1 セッションが複数 role を演じ分ける」排他。CCH は session_id 固有 (`harness_session_register`)、worker/reviewer も別 spawn で「1 セッション = 1 identity」が自然。signal の `to_session` addressing が agmsg の (team,agent) ペアより細かいため lock 不要。
7. **opt-in**。`HARNESS_SESSION_RELAY=monitor` env で配布 default OFF (Phase 53 / 91 の opt-in flag 方針を踏襲)。

team_validation_mode: `subagent` + cross-session + 内部実装再設計の manual-pass。対話で storage 3 層分解 / cross-agent 天井 / actas 要否を多角検証済み。残る未確認点 (signal が双方向 addressing を載せられるか + SessionStart hook から Monitor 常駐起動が可能か) は 93.1 spike を実装可能性 gate として配置。Fleet Mode セッション (`ee40090c`) と signal thread `0280cf7b` で本命の adoring-bhabha 引き取りを合意。
